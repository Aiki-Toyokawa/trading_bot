# RasPi 24/7構成（最新版 / 現行実装準拠）

## 目的
- 目標は「勝率の最大化」より「損失の制御（負けを小さく）」を優先すること。
- AIは最終売買判断をしない。候補抽出・情報構造化・事後分析に限定する。
- 売買執行とリスク管理は Python ルールで固定し、再現性を担保する。

## AIの役割分担
```text
① Finviz: 候補銘柄抽出（スクレイピング）
② Claude: 非構造情報をJSONへ整形（定性情報の構造化）
③ Python: エントリー/イグジット判定（最終意思決定）
④ Alpaca: 注文執行
⑤ Claude: 約定が発生したrunのみログ分析
```

## main.py 実行フロー（run_history保存込み）
1. `.env` 読み込み
2. DB初期化 + ランタイムマイグレーション
   - `trades` 補助カラム追加（`close_reason`, `initial_qty`, `take_profit_done`, `trailing_high_price`）
   - `order_logs.realized_pnl_amount` 追加
   - 同一symbolのOPEN重複自動補正
   - `uq_trades_symbol_open`（OPEN中同一symbol一意）
3. DB保持期間クリーンアップ（signals / position_snapshots）
4. Client Setup（Alpaca / Claude / Finviz）
5. 接続状態判定を作成
   - `connectivity.alpaca_api` / `anthropic_api` / `finviz_web`
   - 値は `OK | NG | UNKNOWN`
6. Market Gate（市場時間ガード）
7. Sync Flow（遅延約定反映・手動売却復元・数量同期）
8. Exit Flow
9. Entry Flow
10. Stats集計
11. 約定が1件以上あるrunのみ Claudeログ分析・保存
12. 実行結果を `run_history` に必ず保存（成功/スキップ/例外問わず）

## Debug/Testモード
`DEBUG_EXECUTION_MODE`
- `0`: 通常実行（判定 + 注文送信）
- `1`: 判定のみ（売 + 買）
- `2`: 売判定のみ
- `3`: 買判定のみ

`DEBUG_FORCE_RUN_WHEN_MARKET_CLOSED=False` でも、`DEBUG_EXECUTION_MODE` が `1/2/3` の場合は閉場時に判定ログを出す（注文は送信しない）。

## 市場時間ガード
- `MARKET_HOURS_GUARD_ENABLED=True`
- Alpaca clock が取得できる場合は clock を優先
- clock失敗時は JST fallback
  - 開始: `22:30`
  - 終了: `05:00`
  - セッション曜日: 月〜金（開始日の曜日基準）

## Entryフロー（現行）
1. 日次実現損益を確認（`DAILY_LOSS_LIMIT` 到達で停止）
2. 保有数を確認（`MAX_POSITIONS`）
3. 未約定BUY注文があるsymbolを除外（重複発注防止）
4. Finviz候補取得（`FINVIZ_CANDIDATE_LIMIT`）
5. Alpaca日足取得・指標計算（SMA/RSI/ATR/出来高）
6. Claude JSONスコア取得
7. Pythonロジックで最終判定（`evaluate_entry_candidate`）
8. 資金配分（均等 or score weighted）
9. 発注（通常モードのみ）

### Entry判定の主要条件
- 価格帯: `MIN_PRICE_USD <= price <= MAX_PRICE_USD`
- 出来高: `avg_volume >= effective_min_avg_volume`
- `avoid=true` は除外
- `event_risk=high` は除外
- トレンド: 基本 `SMA50 > SMA200`
- RSI: 動的上限（通常65, 強トレンド75）+ 上昇方向
- ATR比率: `MIN_ATR_PCT <= atr_pct <= MAX_ATR_PCT`
- 最終スコア: `ENTRY_MIN_SCORE` 以上
- 実発注スコア: `ENTRY_ORDER_MIN_SCORE` 以上（paper緩和時は `TEST_ENTRY_ORDER_MIN_SCORE`）

### Paper緩和（任意）
`USE_PAPER_ACCOUNT=True` かつ `TEST_ENTRY_RELAX_MODE=True` のとき:
- 出来高閾値に `TEST_MIN_AVG_VOLUME_FACTOR` を適用
- `trend_not_up` / `rsi_condition_failed` を条件付き救済
- 1runの新規発注上限を `TEST_MAX_NEW_ORDERS_PER_RUN_OVERRIDE` に切替

## Exitフロー（現行）
1. OPEN trade を取得
2. Alpaca保有を取得（Alpaca無効時はシミュレーション値で判定のみ継続）
3. 未約定SELL注文symbolを除外（重複発注防止）
4. `decide_exit()` の結果で処理

### Exit判定順序
1. 損切り: `current_price <= entry_price * STOP_LOSS_RATE`（-3%）
2. 急落即時退出: 前日終値比 `<= CRASH_EXIT_FROM_PREV_CLOSE_PCT`（-6%）
3. 半利確: `current_price >= entry_price * TAKE_PROFIT_RATE`（+6%で半分売却）
4. 半利確後は高値追随トレーリング
   - 通常: `TRAILING_RATE=0.97`（高値-3%）
   - 急落警戒: `TRAILING_RATE_TIGHT=0.985`（高値-1.5%）
   - 強トレンド: `0.965` / `0.955` に段階緩和
   - 最低利益フロア: +1%（強含み益時は+3%へ引き上げ）

## Syncフロー（不整合吸収）
- 未確定注文を照合し、遅延約定をDBへ反映
- AlpacaにあるがDBにない保有を取り込み
- DBにあるがAlpacaにない保有を補正
  - 売却履歴から復元可能なら損益付きクローズ
  - 復元不可なら `CANCELLED`
- API失敗時の疑似約定は行わない

## ログ分析の実行条件
run内の約定件数（BUY/SELL/同期反映）が `>0` の場合のみ `Claude.analyze_logs()` を実行。
約定0件の場合:
```json
{
  "skipped": true,
  "reason": "no_filled_orders_in_this_run"
}
```

## Finviz条件連動（厳密運用）
`infra/finviz_scraper.py` は `app_settings.py` の値に連動してURL生成。
- 出来高下限: ceil（設定値以上を保証）
- 価格下限: ceil（設定値以上を保証）
- 価格上限: floor（設定値以下を保証）

## タイムゾーン運用
- `TIMEZONE_MODE = UTC | JST`（互換で `JTC` は JST扱い）
- ランタイムログ表示は `runtime_logger.py` で一括反映
- 日次集計（day/today）は SQLite時差補正で反映

## effective_min_avg_volume の算出
`get_effective_min_avg_volume(ALPACA_DATA_FEED)` は以下を順に反映:
1. 基本値 `MIN_AVG_VOLUME`
2. IEX補正（`IEX_VOLUME_ADJUSTMENT_FACTOR`）
3. paper緩和（`TEST_MIN_AVG_VOLUME_FACTOR`）

現設定:
- `80,000 x 0.50 x 0.60 = 24,000`

## DBテーブル（要約）
- `signals`: 判定スナップショット
- `orders`: 注文リクエスト/レスポンス
- `order_logs`: 約定イベント（BUY_FILLED / BUY_NOT_FILLED / SELL_FILLED）
- `trades`: ポジション単位の OPEN / CLOSED / CANCELLED
- `position_snapshots`: Exit時の保有スナップショット
- `analysis_reports`: Claude分析レポート
- `run_history`: 各runの実行履歴（status, mode, filled_count, result_json など）

## RasPi運用（現行）
- main定期実行: `trading-bot-run.timer`（180秒）
- 監視Web常駐: `trading-bot-monitor.service`（8787/TCP）
- 詳細手順は `セットアップ.md` を参照

## 運用意図（現状）
- 防御（損失制御）を最優先しつつ、半利確 + 高値追随で収益機会を残す。
- Syncフローで手動介入・遅延約定・DB不整合を吸収する。
- Paperでログを蓄積し、閾値を段階的に本番寄りへ戻す。
