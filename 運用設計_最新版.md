# RasPi 24/7構成（最新版 / 現行実装準拠）

## 目的
- 最優先は「勝率最大化」より「負けを小さくする」こと。
- AIは最終売買判断をしない。候補選定・情報構造化・ログ分析に限定する。
- 売買執行とリスク管理は Python ルールで固定し、再現性を担保する。

## AI分担
```text
① Finviz: 候補銘柄抽出（スクレイピング）
② Claude: 非構造情報をJSONへ整形（定性情報を構造化）
③ Python: エントリー/イグジット判定（最終意思決定）
④ Alpaca: 注文執行
⑤ Claude: 約定が発生したrunのみログ分析
```

## 実行フロー（main.py）
1. `.env` 読み込み
2. DB初期化 + ランタイムマイグレーション
   - `trades.close_reason` 追加
   - `trades.initial_qty` 追加
   - `trades.take_profit_done` 追加
   - `trades.trailing_high_price` 追加
   - `order_logs.realized_pnl_amount` 追加
   - 同一symbolのOPEN重複を自動補正
   - `uq_trades_symbol_open`（OPEN中同一symbolを一意）作成
3. DB保持期間クリーンアップ（signals / position_snapshots）
4. Client Setup（Alpaca / Claude / Finviz）
5. Market Gate（市場時間ガード判定）
6. Sync Flow（遅延約定反映・手動売却復元・数量同期）
7. Exit Flow
8. Entry Flow
9. Stats集計
10. 約定が1件以上あるrunのみ Claudeログ分析・保存

## Debugモード仕様（市場閉場時の判定ログ対応）
`DEBUG_EXECUTION_MODE`
- `0`: 通常実行（判定 + 注文送信）
- `1`: 判定のみ（売 + 買）
- `2`: 売判定のみ
- `3`: 買判定のみ

`DEBUG_FORCE_RUN_WHEN_MARKET_CLOSED=False` でも、`DEBUG_EXECUTION_MODE` が `1/2/3` の場合は、閉場時でも判定ログを出す（注文送信はしない）。

## 市場時間ガード
- `MARKET_HOURS_GUARD_ENABLED=True`
- Alpaca clock が取得できる場合はそれを優先
- clock失敗時はJST fallback
  - 開始: `22:30`
  - 終了: `05:00`
  - セッション曜日: 月〜金（開始日の曜日基準）

## Entryフロー（現行ロジック）
1. 日次実現損益を確認（`DAILY_LOSS_LIMIT` 超過なら停止）
2. 現在保有数を確認（`MAX_POSITIONS`）
3. 既存未約定BUY注文を検出し重複発注を回避
4. Finviz候補取得（`FINVIZ_CANDIDATE_LIMIT`）
5. 各候補のAlpaca日足取得（不足データは除外・件数をログ化）
6. 指標算出（SMA50/200, RSI, ATR, 出来高）
7. Claude JSONスコア取得
8. Pythonルールで最終判定
9. 資金配分（score weighted対応）
10. 発注（通常モードのみ）

### Entryの主判定条件
- 価格帯: `MIN_PRICE_USD <= price <= MAX_PRICE_USD`
- 出来高: `avg_volume >= effective_min_avg_volume`
- `avoid=true` は除外
- `event_risk=high` は除外
- トレンド: 基本 `SMA50 > SMA200`
- RSI: 動的上限（通常65, 強トレンド75）+ 上昇方向条件
- ATR比率: `MIN_ATR_PCT <= atr_pct <= MAX_ATR_PCT`
- 最終スコア: `ENTRY_MIN_SCORE` 以上
- 実発注スコア: `ENTRY_ORDER_MIN_SCORE` 以上

### paperテスト時の緩和
`USE_PAPER_ACCOUNT=True` かつ `TEST_ENTRY_RELAX_MODE=True` のとき:
- 出来高閾値に `TEST_MIN_AVG_VOLUME_FACTOR` を適用
- `trend_not_up` / `rsi_condition_failed` を条件付き救済
- 実発注スコア閾値を `TEST_ENTRY_ORDER_MIN_SCORE` に切替
- 1runの新規発注上限を `TEST_MAX_NEW_ORDERS_PER_RUN_OVERRIDE` に切替

## Exitフロー（現行ロジック）
1. OPEN trade を取得
2. Alpaca保有を取得（無効時は疑似ポジション）
3. 既存未約定SELL注文を検出し重複発注回避
4. 銘柄ごとに `decide_exit()` 判定

### Exit判定順序
1. 損切り: `current_price <= entry_price * STOP_LOSS_RATE`（-3%）
2. 急落即時退出: 前日終値比 `<= CRASH_EXIT_FROM_PREV_CLOSE_PCT`（-6%）
3. 半利確: `current_price >= entry_price * TAKE_PROFIT_RATE`（+6%で半分売却）
4. 半利確後:
   - 高値追随（分足高値参照オプションあり）
   - トレーリング通常 `TRAILING_RATE=0.97`（高値-3%）
   - 急落警戒時は `TRAILING_RATE_TIGHT=0.985`（高値-1.5%）
   - 強トレンド時は段階的に `0.965` / `0.955` へ緩和
   - 最低利益フロア（+1%、強含み益時は+3%）

## Syncフロー（重要）
- 未確定注文を照合し、遅延約定をDBへ差分反映
- AlpacaにあるがDBに無い保有を取り込み
- DBにあるがAlpacaに無い保有を補正
  - Alpaca売却履歴があれば損益付きで復元クローズ
  - 復元不可なら `CANCELLED` 化
- API失敗時に疑似約定はしない（`status=error` で扱う）

## ログ分析の実行条件
- run内で `BUY/SELL/同期反映` を含む約定が1件以上発生した場合のみ `Claude.analyze_logs()` を実行
- 約定0件の場合:
```json
{
  "skipped": true,
  "reason": "no_filled_orders_in_this_run"
}
```

## effective_min_avg_volume の計算
`get_effective_min_avg_volume(ALPACA_DATA_FEED)` は以下を順に反映:
1. 基本値 `MIN_AVG_VOLUME`
2. IEX補正（`IEX_VOLUME_ADJUSTMENT_FACTOR=0.50`）
3. paper緩和（`TEST_MIN_AVG_VOLUME_FACTOR=0.60`）

現設定では:
- `80,000 x 0.50 x 0.60 = 24,000`

## Finviz条件の設定連動
`finviz_scraper.py` は `app_settings.py` を直接参照してURL条件を生成。
- 出来高下限: ceilで「設定値以上」を保証
- 価格下限: ceilで「設定値以上」を保証
- 価格上限: floorで「設定値以下」を保証

## Timezone運用
- `TIMEZONE_MODE` は `UTC` / `JST`（互換で `JTC` も `JST` 扱い）
- ログ表示時刻は `runtime_logger.py` が一括反映
- 日次集計（today, day判定）は SQLiteクエリの時差補正で一括反映

## DBテーブル（要約）
- `signals`: 判定結果スナップショット
- `orders`: 注文リクエスト/レスポンス
- `order_logs`: 約定イベント（BUY_FILLED/BUY_NOT_FILLED/SELL_FILLED）
- `trades`: ポジション単位のOPEN/CLOSED/CANCELLED管理
- `position_snapshots`: Exit時の保有状況記録
- `analysis_reports`: Claude分析レポート

## 現運用での意図
- 防御（損失制御）を最優先しつつ、半利確 + 追随トレーリングで収益機会を残す。
- 同期フローで「手動介入」「遅延約定」「DBとAlpacaの不整合」を吸収する。
- paperで判定機会を確保し、ログ蓄積後に閾値を段階的に本番寄りへ戻す。

## RasPi運用メモ
- 常駐は `systemd` 推奨（クラッシュ自動再起動）。
- 実行間隔は `RECOMMENDED_POLLING_SECONDS=45` を基準に、負荷と約定機会で調整。
- 24/7稼働でも実売買は市場時間ガードで制御。
